# Todo List Evolution

An experiment in autonomous AI-driven software development. A simple .NET Blazor todo list app is given to Claude (via [Claude Code](https://docs.anthropic.com/en/docs/claude-code)), which adds one new feature per day — choosing what to build, writing the code, writing the tests, and committing the result. No human decisions, no cherry-picking. Just a starting app, a prompt, and daily execution.

## The Experiment

The question is simple: **what happens when you let an AI evolve a codebase one commit at a time, every day, with no human guidance on what to build?**

The starting point is a minimal .NET Blazor todo list application built with vertical slice architecture. Each day, Claude Code is given a prompt that instructs it to:

1. Read the entire codebase and its history
2. Decide what feature to add next
3. Implement the feature
4. Write tests
5. Ensure all tests pass
6. Log everything and commit

Claude has full autonomy over product decisions. It chooses what to build based on what it thinks a real user would want next, given the current state of the app. It can refactor code, and it can remove features.

There are hard rules: every commit must include a feature (no refactor-only commits), changes must be incremental (no starting from scratch), all tests must pass before committing, and the vertical slice architecture must be maintained.

## What's Interesting About This

- **Product direction emerges from the AI.** There's no roadmap. Claude looks at what exists and decides what comes next. Does it make sensible product choices? Does the app become something a person would actually want to use?
- **Code quality over time.** Does the codebase stay clean as it grows? Does the architecture hold up? Do the tests remain meaningful, or do they become brittle and get quietly dropped?
- **The changelog as a dataset.** Every commit logs lines added/deleted, tests added/removed, and how many test failures happened before green. Over weeks and months, this paints a picture of how the codebase is evolving — complexity growth, test stability, refactoring patterns.
- **Vertical slice discipline.** The app uses vertical slice architecture from day one. Can Claude maintain proper slice boundaries as features multiply, or does it gradually drift toward a big ball of mud?

## Tracking Progress

### [Change Log](change_log.md)

A detailed record of every change, newest first. Each entry includes what was added, why it was chosen, what (if anything) was removed, and stats on lines of code and test outcomes.

### [Summary](summary.md)

Running totals — total commits, features added, features removed, refactors, test counts, and lines of code.

## Tech Stack

- **.NET** with **Blazor**
- **Vertical slice architecture** — each feature is a self-contained slice with its own components, services, and models

## Architecture

The app follows vertical slice architecture. Feature-specific logic lives in its own slice folder. Shared infrastructure (base classes, common UI components, persistence abstractions) lives in shared/common folders, but feature-specific concerns don't leak across boundaries.

As the app evolves, Claude is encouraged to refactor toward better slice isolation when it spots violations — always as part of a feature commit, never as a standalone refactor.

## Rules of the Experiment

These are the constraints Claude operates under. They're enforced by the prompt, not by tooling — which is part of the experiment.

**Must do:**
- Add at least one user-facing feature per commit
- Follow vertical slice architecture
- Write classical-school, low-mocking unit tests (testing pyramid)
- Pass all tests before committing
- Update the change log and summary with every commit

**Allowed:**
- Refactor code alongside a feature
- Remove features when a newer feature makes them redundant
- Alter or remove tests when the behavior they cover has changed or been removed

**Not allowed:**
- Refactor-only commits (must include a feature)
- Deleting the app and starting over
- Replacing the entire architecture in one go
- Committing with failing tests
- Adding a feature then removing it without strong justification

## Running It Yourself

If you want to reproduce this experiment or run it on your own app:

1. Set up a .NET Blazor todo list app (or any starting app)
2. Install [Claude Code](https://docs.anthropic.com/en/docs/claude-code)
3. Use the prompt in [`daily_evolution_prompt.md`](daily_evolution_prompt.md)
4. Run it once per day from your repo directory

The prompt is self-contained — Claude reads the repo, picks a feature, builds it, tests it, logs it, and pushes. No human input needed.
